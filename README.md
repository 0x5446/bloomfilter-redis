<h2 id="背景" data-source-line="5"><a class="markdownIt-Anchor" href="#背景"></a>背景</h2>
<p data-source-line="7">最近有一个项目是点击日志（10亿/天）实时计算，架构上简单来说就是利用flunted去从前端机收集原始日志，然后发给Kafka，Spark消费日志并计算保存结果到Redis。</p>
<p data-source-line="9">Kafka的Producer和Consumer端的配置是异步且保证不丢消息，因此当超时发生时，就可能会导致消息的重发或者重复消费，需要在消费环节保证幂等。Spark消费逻辑主要是根据多个维度进行计数计算，因此，我们需要在计算之前去重来保证不重复计数。</p>
<p data-source-line="11">考虑到去重数据规模很大，为10亿量级，且我们的业务场景允许FP（False-Positive，假阳性，即实际为非重复数据，被误判为重复数据），因此自然而然考虑到用Bloom-Filter（<mark>布隆过滤器</mark>）这个极其节约空间，且时间复杂度也极低的，存在一定的误判（可控）的算法。</p>
<h2 id="bloom-filter" data-source-line="13"><a class="markdownIt-Anchor" href="#bloom-filter"></a>Bloom-Filter</h2>
<h3 id="介绍" data-source-line="15"><a class="markdownIt-Anchor" href="#介绍"></a>介绍</h3>
<p data-source-line="17">布隆过滤器（Bloom filter）是由巴顿.布隆于1970年提出的。它实际上是一个很长的二进制向量和一系列随机映射函数。</p>
<p data-source-line="19">Bloom filter的思想很简单优雅。我们假设有k个hash function和m位bit的向量filter：
处理输入的过程如下：</p>
<ol data-source-line="21">
<li>使用k个hash函数计算hash值；</li>
<li>将每个hash值对m取余，得到k个在filter中的位置；</li>
<li>将这k个位置的bit置为1</li>
</ol>
<p data-source-line="25">判定一个输入是否在filter中的操作如下：</p>
<ol data-source-line="26">
<li>使用k个hash函数计算hash值；</li>
<li>将每个hash值对m取余，得到k个在filter中的位置；</li>
<li>看所有的位置是不是都是1，如果是返回true，否则返回false</li>
</ol>

<h2 id="redis数据结构string的setbit方法" data-source-line="78"><a class="markdownIt-Anchor" href="#redis数据结构string的setbit方法"></a>Redis数据结构String的SETBIT方法</h2>
<blockquote data-source-line="80">
<p><strong>SETBIT key offset value</strong></p>
<p>对 key 所储存的字符串值，设置或清除指定偏移量上的位(bit)。</p>
<p>位的设置或清除取决于 value 参数，可以是 0 也可以是 1 。</p>
<p>当 key 不存在时，自动生成一个新的字符串值。</p>
<p>字符串会进行伸展(grown)以确保它可以将 value 保存在指定的偏移量上。当字符串值进行伸展时，空白位置以 0 填充。</p>
<p>offset 参数必须大于或等于 0 ，小于 2^32 (bit 映射被限制在 512 MB 之内)。</p>
<ul>
<li>可用版本：&gt;= 2.2.0</li>
<li>时间复杂度: O(1)</li>
<li>返回值：指定偏移量原来储存的位。</li>
</ul>
</blockquote>

<h2 id="优化" data-source-line="301"><a class="markdownIt-Anchor" href="#优化"></a>优化</h2>
<p data-source-line="303">上面代码中，每次往Bloom filter中add一条数据，需要请求k次redis，性能都损耗在网络IO上了，我们先将这个环节给优化掉。</p>
<h3 id="redis的pipelining介绍" data-source-line="305"><a class="markdownIt-Anchor" href="#redis的pipelining介绍"></a>redis的pipelining介绍</h3>
<p data-source-line="307">Redis Pipelining可以一次发送多个命令，并按顺序执行、返回结果，节省RTT(Round Trip Time)。</p>
<p data-source-line="309">每个SETBIT都是独立的，之间没有任何联系，没有必要保证其原子性，因此无需采用multi方式，距相关资料查证，采用pipelining的效率提升10倍左右，而multi反而会降低效率。</p>

<h2 id="再优化" data-source-line="398"><a class="markdownIt-Anchor" href="#再优化"></a>再优化</h2>
<p data-source-line="400">刚刚Redis官方文档里面对SETBIT的介绍中有这样一句：</p>
<blockquote data-source-line="401">
<p><mark>bit 映射被限制在 512 MB 之内</mark></p>
</blockquote>
<p data-source-line="403">往回翻看上文中</p>
<blockquote data-source-line="404">
<p>False-Positive-Ratio表（含内存空间占用）</p>
</blockquote>
<p data-source-line="406">可以看到如果m为500亿，Bloom filter的内存空间会占用大约5.82GB，大大查过Redis的bit映射范围限制。</p>
<p data-source-line="408">因此我们需要对该Bloom filter实现做分布式改造，根据m的规模， 构建多个bit表，不同的输入会sharding到对应的bit表。</p>
<h3 id="分布式bloom-filter" data-source-line="410"><a class="markdownIt-Anchor" href="#分布式bloom-filter"></a>分布式Bloom-Filter</h3>
<p data-source-line="412">考虑到单个redis实例的内存是有上限的，我们可以设计两级sharding：</p>
<ol data-source-line="413">
<li>第一级将不同的输入sharding到对应的redis实例</li>
<li>第二级将输入sharding到对应的key上（不同的key代表不同的Bloom filter）</li>
</ol>
